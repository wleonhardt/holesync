# holesync code review — 2026-07-03

Scope: full read of `holesync.py` (v1.3.0), tests, config, systemd units;
verified against the Pi-hole v6 OpenAPI specs (FTL master: `auth.yaml`,
`lists.yaml`, `domains.yaml`, `action.yaml`) and the live primary (FTL v6.6).
Bugs below were **reproduced by running the code**, not just read.

## What holds up (verified against the API spec)

- `POST /api/lists?type=` — type as query param: **correct** per spec.
- Array-form POSTs (`address`/`domain` as arrays): **supported** per spec — batching design is valid.
- `:batchDelete` body shapes (`{item, type[, kind]}`): **correct**.
- Session lifecycle (login → `X-FTL-SID` → `DELETE /api/auth` logout): correct; avoids exhausting API seats. 429 "no free seats" retried with backoff — right call.
- `PATCH /api/config` for `dns.hosts`/`cnameRecords`: the right zero-downtime mechanism.
- `POST /api/action/gravity` streams chunked `text/plain`; treating it as best-effort with a long timeout is acceptable.
- Group remap by name, groups-first ordering, diff-gating, shrink guards, converge-after-slow-write: all sound and well-tested.
- Secrets hygiene: `holesync.conf`/`backups/` gitignored; nothing sensitive tracked.

## Confirmed bugs (reproduced)

### B1 — Example config crashes the parser (P0)
`ConfigParser()` is constructed with defaults: no `inline_comment_prefixes`, and
`BasicInterpolation` on. Consequences, both reproduced:
- Loading `holesync.conf.example` verbatim (as README's install step does) →
  `ValueError: Not a boolean: 'true          ; local A/AAAA records…'` — an
  uncaught traceback, not a clean exit-2.
- A password containing `%` → `InterpolationSyntaxError`.

Fix: `configparser.ConfigParser(interpolation=None, inline_comment_prefixes=(";", "#"))`,
and catch `ValueError`/`configparser.Error` in `load_config` → exit 2.
Add a test that loads `holesync.conf.example` verbatim.

### B2 — `run_gravity` silently defaults ON (P0)
`Options.run_gravity` defaults `False` and README/DESIGN document off-by-default,
but `load_config` uses `gb(sy, "run_gravity", True)`. Any config that omits the
key gets surprise gravity rebuilds — the exact heavy operation DESIGN.md warns
about for weak replicas. Reproduced: key absent → `True`. Fix: default `False`.

### B3 — Replica names containing a dot are mangled (P0)
`name = sect.split(":", 1)[-1].split(".", 1)[-1]` turns `[replica:pihole.lan]`
into name `lan`. Reproduced. Also: unrecognized sections (typos like
`[replica pihole-02]`) are silently ignored. Fix: only strip the
`replica:`/`replica.` prefix once; warn/error on unknown sections.

### B4 — `--check` masks replica failures (P0)
`drifted` only counts `r.ok` replicas; check mode returns via
`if drifted: 10 else: 0` without consulting failures. An unreachable or
auth-failing replica during `--check` prints "all replicas in sync", exit 0 —
worst case for the monitoring use-case `--check` exists for. Fix: in check
mode, return the failure code when any replica errored.

### B5 — Backup rotation mixes DNS and collection backups (P1)
DNS backups rotate on prefix `"<name>-"`, which also matches
`<name>-domains-…`, `<name>-clients-…` etc. All kinds share one
`backup_keep` pool, and DNS backups (digits sort before letters) get evicted
first. Fix: rotate on an exact pattern (`name-<stamp>.json` /
`name-<kind>-<stamp>.json`) per kind.

## Weaknesses (not reproduced-crash, but real)

- **W1 — Silent write failures (P1).** `write_item` swallows every error and
  callers ignore the returned status. A 400 (e.g. regex rejected by the
  replica) surfaces only as a generic "did not converge" two reads later. Log
  method+path+status+API error message on any non-2xx.
- **W2 — Malformed replica item kills the whole run (P1).** `plan_collection`
  does `it["type"]`/`it["domain"]` on replica-returned items; a missing key is
  an uncaught `KeyError` that aborts remaining replicas. Catch `Exception` per
  replica in `process_replica` (report as failure, continue).
- **W3 — `urllib.parse` used but never imported (P1).** Works only because
  `urllib.request` imports it as a side effect. One stdlib refactor away from
  `AttributeError`. Add `import urllib.parse`.
- **W4 — Lockfile issues (P2).** `open(path, "w")` truncates the holder's PID
  before attempting the lock; unlink-after-unlock has the classic flock race;
  default `/tmp` + systemd `PrivateTmp=true` means the unit's lock can't
  exclude a manual CLI run. Fix: open `"a"`, don't unlink, default lock to
  `/var/lib/holesync/` (or `/run`), drop the fh on failure.
- **W5 — `ssl._create_unverified_context` (P2).** Private API. Use
  `ssl.create_default_context()` + `check_hostname=False` + `CERT_NONE`.
- **W6 — Verify/rollback flapping risk (P2). → DROPPED (2026-07-03).** The worry
  was: if FTL canonicalizes on write (IPv6 compression, hostname case), every run
  would write → verify-fail → roll back forever. A live probe on the replica
  (FTL v6.6) disproved the premise: a written `fd00:0:0:0:0:0:0:99
  PROBE.Holesync.Test` reads back byte-identical. No canonicalization ⇒ no
  flapping ⇒ no code needed. Not implementing (speculative complexity).
- **W7 — No rollback for gravity collections (doc gap).** DNS sync rolls back;
  collection sync only converges. Backups exist but there's no restore path.
  Either document "restore = manual from backup JSON" or add `--restore`.
  DESIGN.md leans "not holesync's job" — documenting is enough.

## API capabilities worth adopting

- **App passwords (`webserver.api.app_pwhash`).** Spec: sessions require TOTP
  when 2FA is enabled — holesync's plain-password auth would break. App
  passwords bypass 2FA and are the documented automation path. No code change
  needed; document as the recommended credential (+ `password_file`).
- **`GET /api/info/version` pre-flight (optional).** One cheap GET to log
  source/replica FTL versions; warn on mismatch. Aids debugging cross-version
  drift; skip if it feels like creep.
- Teleporter, `restartdns`, direct `gravity.db` access: correctly rejected in
  DESIGN.md — no change.

## Doc nits (P3)

- README cron comment says "hourly" over a `*/30` line.
- README sample shows `max_changes = 25`; code + example file default 100.
- README should state: with 2FA enabled, use an app password (see above).
